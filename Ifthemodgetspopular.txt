# If The Mod Gets Popular: Horizontal Scaling Strategy

If the mod ever hits the point where 20,000+ players are online at the exact same time, a single Free Hugging Face space might start to reach its maximum TCP connection limit. At that point, here is exactly how we scale to handle millions of players for free:

## 1. Multiple Hugging Face Spaces
You can clone your Hugging Face space to create multiple identical backend servers (e.g., `gtsim-server-A.hf.space` and `gtsim-server-B.hf.space`). 

## 2. Client-Side Load Balancing
We would update the Java Minecraft Mod to look at the IP address of the Minecraft server the player is joining, and use a math function (like hashing) to automatically pick a Hugging Face Space.
- For example: Everyone who plays on `mcpvp.club` gets sent to Space A. Everyone who plays on `hypixel.net` gets sent to Space B. 
- This ensures that players who are standing next to each other in Minecraft are always connected to the same Hugging Face Space, so they can see each other's music seamlessly.

## 3. Peer-to-Peer (P2P) Server Clustering
If you want people on Space A to see the global leaderboard or states from Space B, we do NOT need to buy a "Middle Man" server (like Redis).
Because every Hugging Face space has a public URL, we can write a tiny script in Python that tells Server A to open a direct WebSocket connection to Server B!

Because we already built the **"3-Second Batching Optimization"**, Server A won't spam Server B. Every 3 seconds, Server A will simply gather up any new songs, bundle them into a single tiny JSON message, and send it to Server B. Server B adds those songs to its own list and broadcasts them to its players.

This creates an infinitely-scalable, decentralized "Mesh Network" that costs $0 to run!
